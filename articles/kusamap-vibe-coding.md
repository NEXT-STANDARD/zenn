# はじめに

こんにちは、フェニックスです。

人間というのは不思議なもので、何かを作りたいと思った瞬間から、もう作り終わった気分になっている。
企画書を書いている時が一番楽しくて、実装が始まると途端に現実の重さに押しつぶされる。
そういう経験、あなたにもあるでしょう。

今回は「草マップw」という位置情報アプリを作った話をする。

残りは「本当にこれで動くのか？」という疑心暗鬼との戦いだった。

まるで他人の夢の中に迷い込んだような、そんな感覚である。

## 草マップwとは？

「草マップw」（Kusamap W）は位置情報を活用したWebアプリだ。

「草」というのはインターネットミームで「笑う」という意味。
日本のネット文化では「www」が笑いを表し、それが草に見えることから「草」＝「笑い」という意味で使われている。

なぜ「w」が笑いなのか。なぜ笑いが草なのか。
考えてみれば不思議な話だが、言葉というのはそういうものだ。
意味は後からついてくる。

**公開URL**: [https://www.vanishing-vision.com/](https://www.vanishing-vision.com/)

### コンセプト：「この街に、草生やそうぜ。」

草マップwの核心は、**ソウルライクゲームのメッセージ機能を現実世界に実装した**ことにある。

エルデンリングをやったことがある人なら分かるだろう。
ダンジョンの入り口に「この先、崖」と書いてあるあの感じ。
見知らぬ誰かが、見知らぬ誰かのために残したメッセージ。
直接会うことはない。会話もしない。でも確かにそこに誰かがいた。

| ゲーム（エルデンリング等） | 草マップw |
|---------------------------|-----------|
| ダンジョンに先人のメッセージ | 街角に誰かの投稿 |
| 「この先、崖」の警告 | 「この店、激混み」の共有 |
| 評価で残る/消える | 「w」で延命/消滅 |
| 見知らぬ冒険者の助け合い | 見知らぬ街の人との繋がり |

現実世界を冒険するすべての人へ。
誰かが残した「草」を見つけ、価値があれば「w」を送る。
あなたの一言が、次に来る誰かの助けになる。

ゲームの中では何度死んでもやり直せるが、現実はそうもいかない。
だからこそ、先人の知恵は貴重なのだ。

## 草マップwの主な特徴

### 1. エフェメラル（消える）メッセージ

草マップの根底にある価値観は「**情報の価値は鮮度に比例する**」だ。

- 投稿は**24時間で自動消滅**
- 「w」リアクションで延命可能
- **100個以上の「w」**で永久保存（大草原）

位置情報と結びついた情報は、その瞬間・その場所でこそ意味を持つ。
昨日の「ここ混んでる」は今日のノイズでしかない。

世の中のSNSは、すべてを残そうとする。
まるで人類の記憶をすべてアーカイブしなければならないかのように。
だが考えてみてほしい。あなたが3年前に食べたラーメンの写真、誰が見返すのか。

| 従来のSNS | 草マップ |
|-----------|----------|
| 蓄積型（すべて残す） | 淘汰型（価値あるものだけ残る） |
| 古い情報が埋もれる | 古い情報は自動消滅 |
| ユーザーが取捨選択 | システムが鮮度で判断 |

消えることを前提にした設計。
これは怠慢ではない。思想だ。

### 2. 草の成長ライフサイクル

投稿された「草」は時間とともに状態が変化する。
まるで本物の植物のように。

| 時間 | 状態 | 表示 |
|------|------|------|
| 0-6h | 🌱 芽生え | 明るい緑 |
| 6-18h | 🌿 成長中 | 草色 |
| 18-24h | 🥀 枯れかけ | 黄緑 |
| 100+ w | 🌳 不滅 | 深緑 |

芽生え、成長し、枯れていく。
人間の一生も、投稿も、同じようなものかもしれない。
違いは、投稿には「100w」という不死への道が用意されていることくらいだ。

### 3. 位置情報と「行く価値」

- **地図ページでどこからでも投稿を閲覧可能**
- ただし**半径15m以内**に行くと、画像・動画・コメントなど詳細情報が解放される
- 現実世界を「探索」する動機付け

当初は「半径15m以内のみ閲覧可能」という制限をかけていた。
が、それはさすがに不便すぎた。
理想と現実の狭間で、我々は妥協することを学ぶ。

今は地図上でどこからでも投稿の存在は見える。
ただし、実際にその場所に行かないと、画像も動画もコメントも見られない。

「見える」と「分かる」は違う。
遠くから眺めるだけでは、本当のことは分からない。
現地に足を運んで初めて、その場所の「草」の本質が見えてくる。

これは不便さではなく、**行く理由を作る設計**だ。
スマホの画面越しに世界を消費するのではなく、実際に歩いて、その場所に立つ。
そういう体験を促したかった。

### 4. 「w」による価値の証明

消えゆく投稿に他者が価値を認めた時だけ延命する。
**100w到達 = コミュニティが認めた普遍的価値**。

自己評価ではなく、**集合知による選別**。

「いいね」という言葉は使わない。「w」だ。
笑い。共感。そういうもの。
真面目な顔で「価値を評価します」と言われるより、よっぽど人間らしい。

### 5. ゲーミフィケーション

- 1日の投稿上限（レベルで増加）
- 140文字制限
- 「延命の水」などの課金アイテム

投稿に上限があるのは、無限に投稿できると人は堕落するからだ。
制限があるからこそ、何を書くか考える。
Twitterが140文字で始まったのも、そういう理由があったはずだ（今は知らないが）。

### 6. LINE Mini App として動作

- LINE認証のみでログイン可能
- LINEユーザーが気軽に地域の「今」を共有できる

また新しいアカウントを作るのか、とうんざりした顔をしなくていい。
LINEがあれば使える。それだけ。

## バイブコーディングとは？

バイブコーディングは、AIに「委ねる」プログラミング手法だ。

OpenAIの共同創業者であるアンドレイ・カルパシー氏が提唱した概念で、
コードの細部を自分で書くのではなく、AIに自然言語で指示を出して実装を委ねるスタイル。

要するに「こういうの作って」とAIに頼んで、出てきたものを「うん、いいね」と言うだけ。
なんだか詐欺師になった気分だが、実際に動くものができるのだから文句は言えない。

今回の開発では、Claude Codeを使ってこのバイブコーディングを実践した。

## 開発の流れ

バイブコーディングでの開発は、従来のコーディングとは大きく異なる。
というより、「開発」という言葉自体がしっくりこない。
「対話」と言った方が近いかもしれない。

### 1. 要件をAIに伝える

まず、作りたいアプリの概要をClaude Codeに説明した。
- 位置情報を扱うアプリであること
- LINEログインを使いたいこと
- Supabaseをバックエンドに使うこと

「こういうの作りたいんだけど」と友人に相談するような感覚だ。
ただし、この友人は一切疲れを知らず、文句も言わない。

### 2. AIが提案・実装

Claude Codeが適切なファイル構成やコードを提案・生成してくれる。
TypeScriptの型定義からTailwind CSSのスタイリングまで、一貫して対応。

こちらがやることは、出てきたものを眺めて「ふむふむ」と頷くことくらいだ。

### 3. 確認・検証・修正

AIが生成したコードを確認し、動作検証を行う。
問題があれば、その内容をAIに伝えて修正を依頼。

「ここ、動かないんだけど」と言えば直してくれる。
人間の部下にこれをやったらパワハラで訴えられるところだ。

このサイクルを繰り返すことで、アプリが完成していく。

## 開発時間の内訳

総開発時間は**6時間**だったが、内訳は興味深いものだった。

| 作業内容 | 時間 |
|---------|------|
| コーディング（AI生成含む） | 約2.5時間 |
| 確認・検証作業 | 約3.5時間 |

**コーディング自体は半分以下の時間**で済んだ。

時間がかかったのは、むしろ確認・検証作業だ。
- AIが生成したコードが正しく動作するか
- セキュリティ的に問題ないか
- UIが意図通りになっているか

結局、AIを信用していないのだ。
いや、信用してはいけないのだ。
自分で書いていないコードを盲目的に信じるほど、私は楽観主義者ではない。

バイブコーディングは「コードを書かない」のではなく、
**「コードを書く時間を最小化し、検証と意思決定に集中する」**開発手法だと感じた。

## 苦労したこと

正直に言うと、コーディング自体で苦労したことは**特にありませんでした**。

Claude Codeが適切なコードを生成してくれるため、
従来のように「この書き方で合っているか？」「エラーの原因は何か？」と悩む時間が大幅に減った。

苦労がないというのは、喜ぶべきことなのか、それとも何かを失っているのか。
昔のプログラマーは、エラーと格闘することで成長したものだ。
今の私は、エラーが出たら「直して」と言うだけ。

これを進歩と呼ぶのか、退化と呼ぶのか。
正直、よく分からない。

ただし、**確認作業の多さ**は想定以上だった。
- 生成されたコードの動作確認
- エッジケースのテスト
- 各機能の連携確認

これらを丁寧に行う必要があり、結果として開発時間の半分以上を占めた。
AIは書けても、責任は取ってくれない。

## バイブコーディングのメリット・デメリット

### メリット
- **開発速度が圧倒的に速い** - 6時間でアプリが完成
- **技術的なハードルが下がる** - 細かい文法やAPIの使い方を覚えていなくてもOK
- **試行錯誤が容易** - 「こうしたい」と伝えるだけで別のアプローチを提案してもらえる

### デメリット
- **検証作業は省略できない** - AIが生成したコードを盲目的に信じるのは危険
- **要件を明確に伝える力が必要** - 曖昧な指示では期待通りの結果が得られない
- **デバッグ時の理解が浅くなりがち** - 自分で書いていないコードのデバッグは難しい場合がある

メリットとデメリットを並べてみると、結局のところ「便利だけど万能じゃない」という当たり前の結論に落ち着く。
世の中そんなものだ。

## 余談：ドメイン「Vanishing Vision」の由来

少し余談だが、草マップwのドメイン「vanishing-vision.com」には深い意味がある。
まあ、後付けと言われればそれまでだが。

### Xのインディーズアルバムから

「Vanishing Vision」は、日本のロックバンド**X（現X JAPAN）**のインディーズアルバムのタイトルだ。

このタイトルには面白い文字遊びが隠されている：

- **V**anishing **V**ision の頭文字「V」と「V」
- VとVを**横に並べる**と → **W**
- VとVを**上下反転させて縦に並べる**と → **X**
- アルファベット順でも **V → W → X** と並んでいる

YOSHIKIがそこまで考えていたかは知らない。
でも、こういう偶然の符号に意味を見出すのが人間という生き物だ。

### アプリのコンセプトとの一致

「Vanishing Vision」は「消えていくイメージ」という意味だが、これが草マップwのコンセプトと見事にマッチしていた。

**草マップwの仕組み：**
- 投稿は**24時間で消える**仕組み
- いいねの代わりに「**W**」を付けてもらえる
- **Wがたくさん付くと殿堂入り**し、24時間経っても消えない

つまり、残っている投稿は：
1. **新しい最新の情報**か
2. **多くの人に支持された優良な投稿**

のどちらかということになる。

消えていく情報の中で、本当に価値のあるものだけが残る。
この「Vanishing Vision」という言葉がアプリのコンセプトにぴったりだったので、このドメインを取得した。

X JAPANのファンに怒られないことを祈るばかりだ。
