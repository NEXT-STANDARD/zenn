# はじめに

こんにちは、フェニックスです。

「テチテチ」というiOSアプリを個人で作って、先日App Storeに公開しました。
ペットの縦動画SNSです。TikTokみたいにスワイプすると、ペットの動画だけがひたすら流れてくる。本当にそれだけのアプリです。

この記事は、なぜこれを作ったのか・どう設計したのか、という話です。
技術選定の判断と、アルゴリズムを丸ごと公開することにした理由と、App Store審査という最後の壁の話を書きます。最後に、公開後のユーザー数というシビアな現実にも触れます。先に言っておくと、夢のある数字は出てきません。

---

# なぜ「ペット動画だけ」なのか

TikTokもInstagramも、僕はだいたいペット動画を見るために開いています。
柴犬がテチテチ歩く動画、猫がへそ天で寝てる動画、うさぎが草をもしゃもしゃ食べる動画。見ていると本当に癒される。

ただ、ひとつ困っていることがありました。
ペットの動画を見たいのに、3本に1本くらいの頻度で政治のニュースや炎上の動画が挟まってくる。アルゴリズムが次に何を出してくるか分からないので、「ただ癒されたい」という気持ちで安心してスワイプできない。

写真中心のペットSNSはあります。でも、動画を縦画面いっぱいに、滑らかに、スワイプだけで楽しめる場所は、2026年になっても見当たらなかった。

「無いなら作るか」と思ってしまったのが運の尽きです。

このへんの「思ってしまった」は危険な信号で、だいたい数ヶ月の労働に化けます。化けました。

---

# 名前の話

「テチテチ」というのは、小さな動物がちょこちょこ歩く様子の擬態語から取りました。
子犬とか、ハムスターとか、あの「テチテチ」っていう感じの歩き方です。

アプリの中では用語も全部この世界観で統一しています。

| 機能 | サービス内の呼称 |
|---|---|
| 投稿する | **テチる** |
| いいね | **肉球**（タップで肉球スタンプがポンと出る） |
| フォロー | **ついてく** |
| フォロー中フィード | **ついてく中** |
| 全体フィード | **みんな** |

「いいね」を「肉球」と呼ぶ、みたいな細部はどうでもよさそうに見えて、動詞化がバイラルの種だと思っているので、ここは最初から決め打ちでやりました。「テチった？」って言ってもらえたら勝ち、くらいの気持ちです。

まだ誰にも言われていません。

# コアバリューは「滑らかさ」だけに全振りした

機能をモリモリにする誘惑はありました。DMとか、ライブとか、ストーリーズとか。
でも全部やめて、**フィードの滑らかさ**ひとつにコアバリューを絞りました。

具体的な数字で決めています。

- 縦スワイプは **60fps** で動く
- 動画は再生開始まで **2秒以内**

「テチテチ歩く動きが、スクリーンの中でも途切れずに続いてほしい」というのがアプリ名の由来でもあるので、ここが滑らかじゃないと存在意義がない。なので、AVPlayerの先読みとメモリ管理に一番時間をかけました。

やったことはシンプルで、

- `AVPlayer` をプール化して、表示中の前後2〜3件だけ `AVPlayerItem` を先読みする
- 画面外に出たプレイヤーは pause + 解放してメモリを抑える
- ループ再生は `AVQueuePlayer` + `AVPlayerLooper`
- ページングは iOS 17 の `scrollTargetBehavior(.paging)`

地味です。でもSNSのフィードは、この地味さがそのまま体験の質になる。スクロール中に重い処理を1個でも載せると、すぐカクつく。「スクロール中に何もしない」を死守するための実装、という感じでした。

# 設計判断その1：Firestoreを捨ててPostgresにした

最初、バックエンドはFirebase（Firestore）で組むつもりでした。前のプロジェクトでも使っていたし、慣れていたので。

でも、SNSのフィードを真面目に考えた瞬間に詰まりました。

「ついてく中」フィード——つまりフォローしている全ペットの投稿を時系列で出す、という当たり前の機能。これがNoSQLだと地獄なんです。

- フォロー中のID一覧を取る
- そのIDを `in` クエリのチャンクに分割する（Firestoreは `in` が10件までとかの制限がある）
- 各チャンクの結果をクライアント側でマージして時系列ソートする

いわゆる fan-out 問題です。SNSの宿命みたいなやつ。

これがPostgresだと、**JOIN 1本で終わります**。

| | Firestore（NoSQL） | Postgres |
|---|---|---|
| ついてく中フィード | fan-out + `in`チャンク + クライアントマージ | JOIN 1本 |
| いいね数の集計 | サーバレス関数でカウンタ更新 | SQLトリガで完結 |
| ハンドルの一意性 | 予約用の専用処理が必要 | `UNIQUE`制約だけ |

iOSネイティブで作ると決めた時点で、FlutterFire（Flutter用のFirebase SDK）の利点も消えていたので、思い切ってSupabase（Postgres）に全面移行しました。

結果、サーバレス関数の数が激減しました。フィードは全部RPC（Postgres関数）に寄せて、クライアント側では `in` リストもマージも一切組まない。サーバのJOINに全部任せる。

「慣れているから」でFirestoreを選び続けていたら、たぶん今もfan-outのデバッグをしていたと思います。

# 設計判断その2：動画だけCloudflare Streamに逃がした

これは趣味プロジェクトなので、インフラの固定費は**月5,000円以下**を目安にしています。常時起動のサーバは持たない。

そうすると、動画の配信が一番の鬼門になります。動画はとにかく重い。Supabase Storageでそのまま配信すると、egress（転送量）の課金で簡単に予算が吹き飛びます。

なので、**動画だけ** Cloudflare Streamに逃がしました。

Cloudflare Streamは、アップロード → HLS変換 → CDN配信、までを一体で提供してくれるサービスです。配信1,000分でだいたい$1。自前で変換パイプラインを組む必要もない。

動画のフローはこうなっています。

1. クライアントが Edge Function（`create-upload-url`）に署名付きアップロードURLを要求する
2. 先に `posts` テーブルへ「処理中」ステータスでレコードを作る
3. クライアントがCloudflareへ直接アップロードする（Supabaseは経由させない）
4. 変換が終わると Cloudflare が Webhook を投げてくる
5. Webhook受信用の Edge Function（`stream-webhook`）が署名を検証して、postを「ready」に更新する

ポイントは「動画のバイト列を一度もSupabaseに通さない」ことです。Supabaseはメタデータと権限だけを持ち、重いものは全部Cloudflareに渡す。これで月5,000円の予算内に収まっています。

ちなみに `stream-webhook` は外から叩かれる口なので、HMAC署名検証を必須にして、検証に失敗したら問答無用で401を返します。タイムスタンプも5分しか許容しない。このへんを手を抜くと、知らない誰かに勝手にpostをreadyにされたりするので、地味だけど大事なところです。

# アルゴリズムを公開することにした

ここが、テチテチで一番やりたかったことです。

InstagramやTikTokへの一番大きな不満って、「なぜ伸びたのか・なぜ伸びなかったのかが分からない」ことだと思うんです。一生懸命撮って投稿しても、理由が分からない。アルゴリズムはブラックボックスで、何かを信じてやり続けるしかない。

テチテチは、フィードのスコア計算式を全部公開しています。

```
スコア =
    完視聴率 × 60 pt
  + 肉球率   × 30 pt
  + コメント率 × 10 pt
  − 通報ペナルティ × 50 pt

// 各率 = 該当アクション数 ÷ 再生数
```

これだけです。完視聴率（動画の80%以上を見てもらえた割合）を最重視して、次に肉球（いいね）、コメント。通報には大きなマイナスを置く。

## なぜ完視聴率を一番重くしたのか

再生数やいいね数だけで評価すると、サムネイルで釣る動画や、煽り演出が有利になります。そういうのを上位に出したくない。

完視聴率を最重要にすると、「最後まで見てよかった」と思える動画が上に来やすくなる。テチテチは、ペットがただテチテチ歩いているだけの「何気ない動画」が評価されてほしいSNSなので、この設計にしました。

## 「公開したら攻略されるのでは」問題

アルゴリズムを公開すると、必ず「ゲーム（攻略）されるのでは」と言われます。

でも、完視聴率を上げる方法ってひとつしかないんです。**最後まで見てもらえる動画を作ること**。サムネで釣っても、演出で煽っても、完視聴率は上がらない。むしろ離脱されて下がる。

数字を操作するより、ペットの本物の可愛さを撮るほうが早い。だから、式を公開しても問題ないと思っています。攻略法が「いい動画を撮る」なら、それはもう攻略じゃなくて、ただの正攻法です。

しかも公開しておくと、後から係数を変えたときに「いつ・なぜ変えたか」を更新履歴として残さざるを得ない。自分で自分に透明性を強制する仕組みでもあります。

---

# カウンタを改竄できなくする、という地味な戦い

アルゴリズムを公開する以上、スコアの素になる数字——肉球数・コメント数・フォロワー数——をクライアントから書き換えられたら全部おしまいです。式を公開した意味がなくなる。

なので、カウンタ列はクライアントから一切書けないようにしています。

- 列レベルで `UPDATE` 権限を `REVOKE` する（そもそも書けない）
- 実際の増減は `SECURITY DEFINER` のSQLトリガでやる（肉球が1行INSERTされたら、トリガがカウンタを+1する）
- 全テーブルでRLS（Row Level Security）を有効化する

肉球は `(post_id, user_id)` の複合主キーにして、同じ人が同じ投稿に二重で肉球できないようにDB側で防いでいます。クライアントを信用しない。全部DBの制約とトリガで守る。

このへんは作っていて一番地味で、一番楽しくない部分でした。画面には何も出てこないので。でも、ここを手抜きすると「いいね数、API直叩きで盛り放題のSNS」が完成してしまうので、やるしかない。

---

# App Store審査という最後の壁

実装が終わっても、まだゴールじゃありません。Appleの審査があります。

個人開発のSNSアプリ（=ユーザー投稿コンテンツ＝UGCがある）は、Appleが審査でかなり見てくるポイントがいくつかあります。先回りして潰しておかないと、確実にリジェクトされる。

実際に対応したのはこのあたりです。

- **アカウント削除（ガイドライン5.1.1(v)）**: アプリ内から自分のアカウントを完全に消せること。設定から削除を実行すると、Edge Functionが本人を確認して、認証ユーザーをハード削除 → DBの `on delete cascade` でペット・投稿・コメントが全部消え、さらにStorageの画像とCloudflareの動画本体まで消す。「論理削除でフラグを立てるだけ」はNGで、本当に消えないと通らない。
- **Sign in with Appleのトークン失効**: アカウント削除時に、Apple側のトークンも `revoke` する。これも審査要件。
- **ユーザーのブロック機能（ガイドライン1.2）**: UGCがあるアプリは、嫌なユーザーをブロックできる導線が必須。
- **通報機能**: 不適切な投稿を通報できること。

正直に言うと、「アプリの本体機能」よりこの審査対策のほうに後半の体力を持っていかれました。SNSを個人で出すというのは、半分くらいこの「コンプライアンスの実装」なんだなと痛感しました。

で、結果としては——`v1.0 (build 2)` が **2026年6月13日に審査を通過**して、無事App Storeに公開されました。
build 1ではなくbuild 2なのは、お察しの通り一度リジェクトを食らっているからです。フィードの2枚目の画像が黒くなるバグを審査員に見つけられました。お恥ずかしい限りです。

# 公開された。で、ユーザーは？

ここまで読むと、さぞ立派なものができたように聞こえるかもしれません。

公開後のユーザー数について、正直に書きます。

爆発的には増えていません。

App Storeに並んだ瞬間に1万ダウンロード、みたいな世界は、少なくとも僕のところには来ませんでした。当たり前です。誰も僕のアプリの存在を知らないので。

「作るのは、広めることに比べたら簡単」というのは、個人開発をやると全員が通る悟りだと思いますが、僕も例に漏れず今ここにいます。設計に何ヶ月もかけて、審査対策まで完璧にやって、それでも「知られていない」という一点で、ダウンロード数はゼロからのスタートです。

技術的なハードルが下がった分、勝負はプロダクトの外側——どう知ってもらうか——に完全に移っている。今はそこと向き合っています。

でも、悲観はしていません。
これから何を作るか・何をやめたかも、開発者ノートとして公開し続けるつもりです。アルゴリズムの式も公開しているくらいなので、出し惜しみする秘伝のたれは元々ありません。方向性と理由をオープンにしておいて、一緒に育ててもらえたらと思っています。

# まとめ

- ペット動画「だけ」が流れてくるSNSが無かったので、個人で作ってApp Storeに出した
- コアバリューは「フィードの滑らかさ」だけに全振り（60fps / 再生2秒以内）
- バックエンドはFirestoreを捨ててPostgres（Supabase）へ。SNSのフィードはJOINが正義
- 動画だけCloudflare Streamに逃がして、インフラ月5,000円以下を維持
- フィードのアルゴリズム（計算式）を全部公開した。完視聴率を最重視。攻略法が「いい動画を撮ること」なら公開していい
- カウンタはRLS + 列REVOKE + SECURITY DEFINERトリガで改竄不能に
- 個人SNSアプリは、半分くらいApp Store審査対策（アカウント削除・ブロック・通報）の実装
- 公開したからといってユーザーが湧くわけではない（現実）

うちの子がいる方も、まだいない方も、ペット動画で癒されたい方なら楽しめるはずです。
よかったら、3人目くらいのユーザーになってください。
