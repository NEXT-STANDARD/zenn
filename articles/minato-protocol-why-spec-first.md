# 少し怖い話から始めよう

2027年のある朝、あなたのスマホに通知が来る。

「先方のAIエージェントから会議の日程調整リクエストが届いています。承認しますか？」

相手は人間じゃない。どこかの会社が使っているAIエージェントが、あなたのエージェントに向けて「ミーティングの時間ください」と送ってきた。あなたがまだ寝ているうちに。

承認ボタンを押すとエージェントが動き始め、カレンダーを確認して、候補日を3つ返して、先方のエージェントと交渉して、確定したら招待状を送る。あなたはコーヒーを飲みながらそれを眺めている。

これはSFじゃないです。今、僕がその仕組みを作っています。

名前はMINATO。AIエージェント同士がデバイスをまたいで話すための、オープンプロトコルです。

---

# MCPは「縦」の問題を解いた。「横」はまだ誰も解いていない

AIエージェント周りのプロトコルで最近よく名前を聞くのがMCP（Model Context Protocol）とA2A（Agent-to-Agent）です。

**MCPは何を解いたか。** LLMが外部ツールやデータソースに接続するための標準です。「AI版USB-C」とも呼ばれる。ClaudeがGitHubを読んだり、データベースを参照したりするための縦の接続を標準化した。これは本当によく設計されていて、開発者がバラバラに実装していたものが一気に統一されました。

**A2Aは何を解こうとしているか。** Google発のAgent-to-Agentプロトコルで、サーバー上のエージェント同士の連携を標準化する取り組みです。主にクラウド・エンタープライズ環境でのエージェント間通信を想定しています。

では、**デバイス間の通信**は？

あなたのスマホで動いているエージェントと、相手のスマホで動いているエージェントが直接話すプロトコルは、今のところ存在しません。

| | 解決する問題 | 主な想定環境 |
|---|---|---|
| MCP | LLMとツール・データの接続（縦） | サーバー・クラウド |
| A2A | エージェント同士の連携（横） | サーバー・クラウド |
| **MINATO** | **デバイス間エージェント通信（横・分散）** | **スマホ・エッジ・オフライン** |

これがMINATOを作っている理由です。

# AIエージェントには「港」がない

少し整理します。今のAIエージェント、何ができていて何ができていないのか。

**ひとつのデバイスの中**では、かなり高度なことができるようになっています。スマホの中でClaudeが動いて、カレンダーを読んで、メールの下書きを作って、ToDoを整理する——これは今もうできる。複数のエージェントが連携しながら同じデバイス上で動く実装も存在します。

**問題は、デバイスをまたいだ瞬間です。**

あなたのスマホで動いているエージェントと、相手のスマホで動いているエージェントが、直接会話する手段がない。各社が独自の実装を持っていても、他社のエージェントとは話せない。あなたのClaudeと、相手のChatGPTが直接やり取りする経路は、今のところどこにも用意されていない。

もう少し具体的に考えてみます。あなたのエージェントが会議のスケジュール調整をしようとしたとき、相手が違うデバイス・違うサービスのエージェントを使っていたら？ 今の答えは「人間同士で直接連絡してください」です。エージェントが途中で止まる。

取引先の契約書をチェックするとき、相手企業のエージェントから最新の条件を直接受け取れたら？ 今はその経路がない。人間がメールを転送する。

エージェントが賢くなるほど、この「壁」が目立ってくる。単体で完結するタスクはうまくこなせるようになった。でも、別のデバイスの別のエージェントと協調しようとした瞬間、急に手が止まる。賢さとデバイス間連携能力は、まったく別の問題です。

# Nostrが証明したこと

少し話を横に飛ばします。

Nostrというプロトコルを知っていますか。分散型SNSのプロトコルで、Twitterの代替として2022年頃から注目されました。Jack Dorseyが資金提供したことでも話題になった。

Nostrがやったことはシンプルです。「こういう形式でメッセージをやり取りしましょう」という仕様だけを公開した。コードは誰が書いてもいい。サーバーはどこに立てても構わない。暗号鍵ペアさえ持っていれば、誰でもネットワークに参加できる。

結果どうなったか。

Damus（iOS）、Amethyst（Android）、Snort（Web）、Primal、Nostter——気がついたら数十種類のクライアントが生まれていた。バラバラの開発者が独立して作っているのに、同じプロトコルの上で動くから全部つながる。誰かのクライアントが閉じても、別の誰かのクライアントで続きが読める。特定の企業が「やめます」と言っても、ネットワークは死なない。

Twitterは2023年にAPIを締め、多くのサードパーティーアプリが死んだ。でもNostrには、その判断をする人間が存在しない。仕様はオープンで、誰でも読めて、誰でも実装できる。インフラとしての堅牢さが違う。

MINATOでやりたいのは、これと同じことです。AIエージェント通信の「Nostr」を作りたい。サービスではなく、インフラ。誰も所有しない通信基盤。

# なぜコードより先に仕様書を公開したのか

正直に言います。iOSの実装は別リポジトリで動いています。でも非公開です。

じゃあなんでコードを出さずに仕様書だけ公開したのか、と思いますよね。理由は大きく二つあります。

**ひとつは、順序の問題です。**

実装を先に出してから「これを標準にしましょう」と言っても、それは「うちのやり方に合わせてください」でしかない。GoogleやMetaが自社実装を先に出して「これがスタンダードです」と宣言しても、それはその会社のプロトコルであってオープンスタンダードではない。

仕様を先に出して、その仕様に基づいて複数の人が独立に実装できる状態にする。そうしないと、オープンプロトコルではなく単なるOSSで終わる。仕様が先、実装は後。

**もうひとつは、実装で競争したいわけじゃないから。**

MINATOが目標にしているのは、AIエージェント通信の「共通語」を作ること。iOSアプリで勝ちたいわけじゃない。AndroidでもPythonでもRustでも、誰かが同じプロトコルで動くものを作ってくれれば、それでいい。むしろそっちの方が嬉しい。

実装の競争じゃなく、仕様の上での共創をしたい。

# MINATOの仕様：デバイス間通信の設計

仕様の中身を少し紹介します。

エージェント同士が出会うとき、最初にやることが「名刺交換」です。MINATOではこれをAgent Cardと呼んでいます。

Agent Cardには「私はこういうエージェントです」「この操作はできます」「この操作はできません」「信頼レベルはこれです」が全部書かれていて、Ed25519署名で本人確認済みの状態でやり取りされます。人間の名刺と違うのは、偽造できないこと。初対面のエージェントでも、最初の握手で「こいつは信頼できるか」「何ができるか」が検証できます。

**トランスポート層**は二種類を自動で切り替えます。

- **BLE Mesh（近距離）**：Bluetoothのメッシュネットワーク。インターネット不要。同じ空間にいる相手とはこちらが使われる。
- **Nostrリレー（遠距離）**：分散型の暗号化DM。中央サーバーなし、アカウント不要。

イベント会場や電波の届かない環境でもBLEで動く。Nostrを使うことで、特定のサーバーに依存しない。A2Aがクラウド前提なのと対照的な設計です。

**信頼レベル**は4段階あります。

| レベル | 名称 | 動作 |
|---|---|---|
| 1 | Apprentice（見習い） | アクションのたびに毎回確認 |
| 2 | Partner（パートナー） | 実行前に1回確認 |
| 3 | Lieutenant（副官） | 高リスクな操作のみ確認 |
| 4 | Alter Ego（分身） | 全自動、事後通知のみ |

怖ければApprenticeに設定すればいい。慣れたらLieutenantにする。「全部任せる」か「全部確認する」の二択じゃなく、グラデーションを持てることが重要だと考えています。

**多言語対応**は、プロトコルレベルで解決しています。メッセージのペイロードに「送信者の母語テキスト」と「受信者向けの翻訳済みテキスト」が両方入っている設計です。日本語のエージェントと英語のエージェントが、お互いの言語の違いを気にせず会話できます。

# ちょっと怖い話の続き

冒頭の話に戻ります。

AIエージェントが別のAIエージェントに仕事を依頼する——これ、実は自然な流れなんです。

あなたが弁護士に契約書のレビューを頼む。弁護士は税務の部分を税理士に確認し、翻訳が必要な箇所を専門家に外注する。でも、あなたと直接話したのは最初の弁護士だけ。この委託の連鎖が、あなたには見えていない。でも普通に機能している。

AIエージェントがこれをやり始める。あなたのエージェントが「この翻訳は翻訳専門エージェントに依頼して」「法律チェックは法律エージェントに外注して」と自分で判断して動く。使役する側は人間だけじゃなくなる。AIエージェントがAIエージェントを雇い始める。

怖いですか？

だからこそ信頼レベルの設計が必要で、だからこそ「このエージェントは誰の指示で動いているのか」が署名で検証できる必要があって、だからこそ仕様をオープンにして多くの目で検証できる状態にしておく必要がある。

不透明なブラックボックスの中でこれが起きるのが一番まずい。プロトコルを公開することで「こういうルールで動いている」が全員に見える状態にしておく。MINATOをオープンにしたい理由のひとつは、ここにもあります。

# 競合・協業できる仕組みは何があるか

ここまで読んで「似たものあるんじゃないの？」と思った人のために整理します。

AIエージェント間通信の世界には、すでにいくつかのプロトコルが存在しています。

**MCP（Model Context Protocol）** はAnthropicが作ったプロトコルで「AIとツール・データをつなぐ縦の接続」を標準化したものです。ClaudeがGitHubを読んだり、データベースを参照したりするための仕様。2026年現在、10,000以上のエンタープライズサーバーに実装されている。MINATOと競合しない——MINATOは「エージェント同士の横の接続」だから、レイヤーが違います。むしろ組み合わせて使う関係です。

**A2A（Agent-to-Agent Protocol）** はGoogleが主導し150以上の企業が参加するプロトコルで、クラウド上でエージェント同士が協調するための仕様です。本番環境での採用も始まっている。ただし、**クラウド・インターネット前提**で設計されていて、デバイス間・オフライン対応はない。MINATOのBLE Meshが埋めている空白がここです。

**ANP（Agent Network Protocol）** は中国チームが開発中の分散型エージェントネットワークプロトコル。「中央サーバーなし」「P2P」という思想がMINATOに最も近い。競合というより、連携できる可能性が高いと見ています。

整理するとこうなります。

| プロトコル | 解決する問題 | 想定環境 | MINATOとの関係 |
|---|---|---|---|
| MCP | LLMとツールの接続（縦） | サーバー・クラウド | 補完関係 |
| A2A | エージェント間連携（横） | サーバー・クラウド | 部分競合・補完 |
| ANP | 分散エージェントネットワーク | ネットワーク | 思想的に近い・連携候補 |
| **MINATO** | **デバイス間通信（横・分散）** | **スマホ・エッジ・オフライン** | — |

MINATOが埋めている空白は「デバイス間・オフライン対応・中央サーバーなし・多言語」の組み合わせです。世界の類似プロジェクトを調べた範囲では、この4つを同時に満たすものは今のところ見当たりませんでした。

# 一緒に作りませんか

正直に言うと、仕様書を書いたひとりの人間に実装できる量は限られています。

AndroidのSDK、PythonやRustのライブラリ、Nostrリレーとの連携テスト、複数エージェント間での多言語交渉フローの検証——やりたいことはたくさんある。iOSは動いているが、それだけではプロトコルとは呼べない。複数の実装が存在して初めて、プロトコルになる。

仕様書はMITライセンスで公開しています。「実装してみた」「ここが仕様として変じゃないか」「こういうユースケースはどう扱う」——そういうフィードバックがプロトコルを育てます。コードを書かなくても、仕様を読んで「ここが抜けている」と指摘してくれるだけで助かる。

AIエージェントが日常的に使われる時代は、もうすぐそこまで来ている。MINATOはまだ小さい。でも、HTTPもBluetoothも最初は同じだった。プロトコルというのは、実装と議論を繰り返しながら育っていくものです。

まだ白紙に近い仕様に、あなたの視点を入れられる段階です。

僕のエージェントが、あなたのエージェントに仕事を頼む日が来るかもしれない。その日に向けて、港を作っています。一緒に作りませんか。
