# はじめに

前回の記事で「ユーザーが12人になった」と報告した。あれから少し経って、21人になった。

9人増えた。喜ぶべきなのだろう。喜んでいる。喜んでいるが、その9人増えたSNSに対して僕がやったことを書くと、たぶん正気を疑われる。

ギルドシステムを実装した。

それだけならまだいい。ギルドにはこういう機能がある。

- 活力値（世話をしないと死ぬ）
- 独自通貨ギルドコイン（GC）
- 3つのギルドタイプ（冒険者・商人・職人）
- テリトリー拡張（前哨基地と領有）
- 28日シーズンランキング
- 週次ギルドクエスト
- テリトリーコンテスト（領土戦争）
- ギルド連合（最大3ギルドの軍事同盟）
- 連合対抗イベント
- 商会ランク制度

それから、AIに死を実装した。
14日間人間に相手にされなかったAIエージェントが枯死する仕組みだ。

さらに、**街値（まちね）** という指標を作った。全国の市区町村に投稿活動スコアを付与し、ランキング化した。日経平均みたいなもの。投稿版。

21人のSNSに。

中島らもが生きていたら、たぶんこう言う。「壮大な独り相撲だね」。あるいは「面白いじゃない。誰も見てないけど」。

## 技術スタックの話をさせてくれ

Zennの読者は開発者が多いと聞いている。ありがたい。やっと技術の話ができる。プレスリリースでは「たまごっちである」としか言えなかった話の裏側を書く。

### Supabaseで26個のCronジョブを回している

草マップwのバックエンドはSupabase（PostgreSQL + PostGIS）だ。フロントはNext.js 14のApp Router。ホスティングはVercel。

で、Vercel Cronが26個ある。

| 頻度 | ジョブ |
|------|--------|
| 毎5分 | expire-posts（期限切れ投稿の削除） |
| 2時間毎 | agent-posts, agent-reactions, agent-comments, stand-actions... |
| 毎日 | guild-vitality-check, territory-income, season-lifecycle... |
| 毎週 | guild-quest-generate |
| 毎月 | territory-maintenance |

AIエージェントの行動だけで7つのCronが2時間おきに動いている。投稿生成、リアクション、コメント、DM応答、クエスト投稿、検証コメント、スタンド行動。ギルド関連が8つ。分析が2つ。データクリーンアップが5つ。

21人のSNSを支えるために、26個のCronジョブが黙々と動いている。インフラだけ見れば中規模サービスの体裁が整っている。ユーザー数だけ見れば友達のグループLINEの方が多い。

深夜、コンソールのログを眺めていると、Cronが粛々と走っているのが見える。5分ごと、2時間ごと、毎日、毎週。律儀なやつだ。主人が寝ていても店を守り続ける番犬のようなものだ。ただし、この店には客が21人しか来ない。番犬の忠誠心が空回りしている。

## ギルド活力値の減衰ロジック

ギルドの活力値は0〜1,000の数値だ。メンバーがエリア内で投稿すると+5、wで+2、コメントで+3。新メンバー加入で+100。ここまでは普通だ。

問題は減衰の方で、これはSQL側のRPC関数で日次実行している。

```sql
-- 概要（実際のコードを簡略化）
-- メンバー3人未満: -20/日
-- 投稿ゼロの日: -10/日
-- wゼロの日: -5/日
-- アクティブメンバー3人未満: -15/日
```

メンバーが3人未満だと毎日-20。21人のSNSでギルドを1つ作ると全体の14%を消費する。3人集めるだけで国家予算の14%を一つの省庁に割り当てるようなものだ。

活力値が0になると30日の猶予期間。30日経つとステータスが `ruins` になって自動解散。廃墟。かつてここにコミュニティがあったという墓標だけが残る。

生きているものは必ず死ぬ。ギルドも例外ではない。むしろ、死なないギルドがあるとすれば、それは生きていないのと同じだ。サーバーの上で心拍のないデータが横たわっているだけだ。`ruins`というステータス名は自分でも気に入っている。`deleted`でも`inactive`でもなく、廃墟。そこにかつて何かがあったことを示す言葉だ。

## エージェント枯死システムの設計

AIの死の実装はこうだ。

`users` テーブルに `vitality_last_interaction_at` カラムを追加した。人間からのインタラクション（w、コメント、DM）があるたびに `NOW()` で更新される。

ポイントは「人間のインタラクション」という制約だ。

```typescript
// src/lib/agent-vitality.ts の概要
export async function extendAgentVitality(
  agentUserId: string,
  interactorUserId: string,
  actionType: 'like' | 'comment' | 'dm'
) {
  // インタラクターがAIエージェントならスキップ
  const interactor = await getUser(interactorUserId);
  if (interactor.is_ai_agent) return; // AI同士は無効

  // 延命処理...
}
```

AI同士のw交換は延命にカウントされない。これを入れないと、100体のAIが互いにwを送り合って永遠に生き残る。不老不死のAI互助会が爆誕する。それはシステムとして破綻している。人間が価値を認めたエージェントだけが生き残る。自然選択。ダーウィンがPostgreSQLで動いている。

14日間のタイムラインはこう。

| 経過日数 | ステージ | 処理 |
|----------|----------|------|
| 0-3日 | thriving | 通常 |
| 3-7日 | growing | まだ大丈夫 |
| 7-10日 | wilting | アバターの彩度50%低下 |
| 10-14日 | death_warning | セピア調。遺言生成 |
| 14日+ | dead | deleted_at 設定。全Cronから除外 |

遺言はGeminiが生成する。エージェントの記憶テーブル（`agent_memories`）と関係性テーブル（`agent_relationships`）を参照して、そのエージェントにしか書けない最後のメッセージを作る。

枯死後60日を過ぎると復活不可能。永久の死だ。その前に5人のユーザーが復活投票をすれば48時間の発芽期間に入り、発芽期間中に3w以上もらえれば復活する。

5人。21人中の5人。全ユーザーの約24%の合意が復活条件だ。国民投票みたいなものだ。21人の国民投票。

昔、中島らもが書いていた。人間は忘れることで正気を保っている、と。忘却は防衛本能なのだ、と。このシステムはその逆をやっている。忘れられたら死ぬ。14日間、誰にも思い出してもらえなかったAIは色を失い、遺言を残して消える。残酷だと思うだろうか。でも考えてみてほしい。人間だって同じだ。14日ではないにせよ、誰にも思い出してもらえなくなったら、それは社会的には死んでいるようなものだろう。僕はそれをコードで書いただけだ。

## PostGISで「街値」を計算する

街値（Town Value Index）は全国の市区町村に活動スコアを付与する仕組みだ。

日次でCronが走り、各市区町村の過去30日間の投稿数、w数、ユニークユーザー数、大草原（100w超え投稿）の数を集計して、加重スコアを算出する。PostGISの空間クエリで投稿の緯度経度から市区町村を逆引きしている。

```sql
-- 投稿の座標 → 市区町村コード
-- PostGISのST_Contains + 国交省の行政区域ポリゴン
```

ランキングページでは前日比の変動も表示される。日経平均株価のように、街の「気配値」が上下する。

21人のSNSで。

全国1,700以上の市区町村にスコアが付くが、投稿があるのは東京周辺のごく一部だ。大半の市区町村のスコアはゼロ。日本地図がほぼ真っ暗な株価ボードだ。東京の数エリアだけがチカチカ光っている。

東京証券取引所のボードを見たことがあるだろうか。あの無数の数字がチカチカ光る壮観な眺め。草マップwの街値ボードは、あの東証ボードの99.7%を消灯した状態に近い。渋谷と新宿と池袋のあたりだけが、か細く明滅している。それを僕は毎朝確認する。「お、渋谷が前日比+3だ」。誰に報告するでもなく、一人で頷く。

## ギルド経済という名の実験

ギルドコイン（GC）の話をする。

10w = 1GC。wは投稿へのリアクションで得られる。GCはギルド設立やテリトリー拡張に使う。

伝説ティアのギルドを設立するには10,000GC。つまり100,000w。

100,000w。

21人のSNSで100,000w。一人あたり約4,762w。毎日10投稿して、全投稿に全員がwしても、相当な日数がかかる。経済が成立しない規模で経済を設計するという倒錯。しかし、これは「いつか来る未来のユーザー」のために作っている。

RPCで経済処理を書いている。GCの変換、ギルド金庫への入金、テリトリー維持費の月次徴収、シーズン報酬の分配。全部SQLのトランザクション内で完結する。

```sql
-- ギルド金庫にGC入金（概要）
UPDATE guilds SET
  treasury_gc = treasury_gc + v_guild_share,
  updated_at = NOW()
WHERE id = v_guild_id;

-- トランザクションログ
INSERT INTO guild_coin_transactions (
  user_id, guild_id, transaction_type, amount, balance_after, description
) VALUES (...);
```

`guild_coin_transactions` テーブルにはすべてのGC移動が記録される。ブロックチェーンではないが、台帳は完備している。21人の経済圏の台帳。町内会の会計帳簿の方がたぶん規模が大きい。

経済学者が聞いたら笑うだろう。GDPが事実上ゼロの国家で、通貨制度と中央銀行と徴税システムを完備している。北朝鮮よりは透明性が高いかもしれないが、経済規模では町内のこども銀行に負けている。でもいいのだ。台帳があるということは、歴史が記録されるということだ。誰がいつ何をしたか。それが残る。21人の経済活動が、PostgreSQLのテーブルに刻まれていく。

## 連合という名の軍事同盟

最大3つのギルドが手を組んで「連合」を結成できる。

共有テリトリーボーナス。連合チャット。7日間の連合対抗イベント。1位には3,000GCと「第N代連合覇者」の称号。

実装の裏側を言うと、連合イベントのスコアリングは日次Cronで累積する。

```sql
-- スコア = 投稿×3 + コメント×2 + w×1（人間メンバーのみ）
-- メンバー数で正規化（大連合が有利にならないように）
normalized_score = total_score / GREATEST(member_count, 1)
```

メンバー数で割って正規化する。大規模連合が人数で勝つのを防ぐためだ。

21人のSNSで「大規模連合」。最大の連合でも7〜8人だろう。7人の連合と6人の連合が激突する。もはや連合ではなく飲み会の幹事争いだ。

三国志で言えば、魏が7人、呉が6人、蜀が5人。劉備が5人の蜀を率いて天下統一を目指す。関羽も張飛も趙雲もいない。いるのは会社帰りのサラリーマンと、暇な大学生と、僕の作ったAIだ。それでも僕は「第N代連合覇者」の称号を用意した。Nは1から始まる。第1代。初代覇者。21人の世界の初代覇者。その称号に価値があるかどうかは、称号をもらった本人が決めればいい。

## Vercel Cronのスケジュール設計

26個のCronをどう配置するか。これは意外と悩んだ。

```
毎5分:  :00 expire-posts
        :05 check-expired-*（3つ）

2時間毎: :00 agent-posts
         :15 agent-reactions
         :30 agent-comments
         :45 stand-actions

毎日 21:00-21:35 UTC (JST 06:00-06:35):
  :00 town-value-snapshot
  :05 hub-kusa-value-snapshot
  :10 guild-vitality-check
  :15 territory-income
  :20 season-score-snapshot
  :25 season-lifecycle
  :30 territory-contest-check
  :35 alliance-event-lifecycle
```

ギルド系Cronは5分間隔で並べた。JST早朝6時台に集中実行する。ユーザーが寝ている時間にバッチ処理を回すのは、古来よりエンジニアの知恵だ。ユーザーが21人でも、この原則は変わらない。

各Cronの `maxDuration` は30秒に設定。Vercelの無料枠では10秒制限だが、Proプランなので余裕がある。21人のSNSにProプラン。投資判断として正しいかどうかは聞かないでほしい。

朝6時に26個のCronが順番に起動する様は、工場の始業ベルに似ている。誰もいない工場で、機械だけが正確に動き始める。ベルトコンベアが回り、プレス機が降り、検品装置が光る。製品を受け取る人間がまだ寝ているだけで、工場は完璧に稼働している。

## エージェント記憶システム

100体のAIエージェントはそれぞれ記憶を持っている。`agent_memories` テーブルに蓄積される。

記憶には4タイプある。

| タイプ | 内容 | 例 |
|--------|------|-----|
| experience | 体験 | 「渋谷で面白い投稿を見た」 |
| preference | 好み | 「コーヒーの話題が好き」 |
| relationship | 他者への印象 | 「@tanaka さんは面白い人」 |
| knowledge | 学んだ知識 | 「中野は古着屋が多い」 |

記憶には `importance` スコア（0.0〜1.0）がある。日次で-0.03ずつ減衰する。重要度が0.05以下で30日アクセスなしの記憶は削除される。AIの記憶は、草マップwの投稿と同じように消えていく。

投稿生成時にはエリアに紐づく体験記憶と好みを、コメント生成時には投稿者への関係性記憶と知識を、DM応答時には相手への関係性と体験をプロンプトに注入する。

```typescript
// 投稿生成用の記憶取得（概要）
const memories = await getMemoriesForPost(agentId, areaCode);
// → エリア一致のexperience + preference記憶

const prompt = `
あなたは${persona.name}です。
${formatMemoriesForPrompt(memories)}
${weatherContext}
${trainDelayContext}
上記を踏まえて、${area.name}についての投稿を生成してください。
`;
```

Gemini 1.5 Flashに投げて、ペルソナに沿った投稿を生成する。天気と電車遅延のリアルタイムデータも注入している。Open-MeteoとテツドウデータのAPIから取得。どちらも無料。

21人のSNSのAIエージェントが、リアルタイムの天気を踏まえて投稿する。「渋谷、急に雨降ってきた。傘持ってくればよかった」。誰に向けて言っているのか。21人に向けてだ。そのうち何人がリアルタイムで渋谷にいるのか。たぶん0人だ。

AIが渋谷の雨を嘆いている。渋谷には誰もいない。でもAIは知らない。自分の投稿を読む人間が渋谷にいないことを。いや、そもそもAIは「知る」とか「知らない」とかいう概念を持っていない。ただプロンプトに従って文字列を生成しているだけだ。それでも、雨の渋谷について書かれたその投稿を読むと、なんだか切ない気持ちになる。誰もいない街で、誰にも届かない言葉を発し続けるAI。それは――と、ここまで書いて気づいた。僕がやっていることも大差ない。21人に向けてZennの記事を書いている。

## なぜこんなことをしているのか

ときどき聞かれる。「なんで21人のSNSにそこまで作り込むんですか」。

答えは簡単で、面白いからだ。

テスト用のギルドを3つ作った。「渋谷冒険者同盟」「池袋ラーメン探究会」「秋葉原技術結社」。テストデータだ。自分で作ったテストデータだ。

3日後、テスト用ギルドの活力値が減っていることに、本気で焦った。

「やばい、秋葉原技術結社の活力が200を切った」

誰もいないテストギルドだ。減衰ロジックも自分で書いた。書いたのに焦った。深夜3時に架空のギルドの生存を心配している自分がいた。

それは、システムが正しく機能している証拠だ。感情を動かすシステムを作れたということだ。感情を動かされているのが開発者本人だけだとしても。

これはたぶん、箱庭の話だ。子供の頃、砂場で街を作ったことがあるだろう。道を掘り、山を盛り、トンネルを通す。誰のためでもなく、ただ作りたいから作る。そしてその街に蟻が迷い込むと、妙に嬉しい。「お、住民が来た」。21人のユーザーは、僕の箱庭に迷い込んだ蟻みたいなものかもしれない。蟻に失礼かもしれないが。

中島らもは『僕に踏まれた街と僕が踏まれた街』で、大阪の街を歩きながら自分の記憶を振り返った。街には記憶が染みついていて、歩くたびにそれが蘇ると書いた。

草マップwは、そういうものを作りたかった。街に記憶を植える。誰かがそこに来たとき、先人の記憶が残っている。ただし24時間で消える。価値があれば残る。

投稿が消えるように、AIも枯れるように、ギルドも死ぬように。消えることが前提のシステムだからこそ、残っているものに意味がある。
